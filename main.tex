\section*{Problem Description}
% ===================================================================

New England SteamWorks (NES) dispatches multiple technicians with service vehicles to perform maintenance jobs across a wide geographic region every week.
The planning engine follows a \textbf{rules--first} approach: every scheduling and routing decision is driven by explicit, named business rules rather than opaque numerical optimisation.
The weekly planning task proceeds in two deterministic phases:

\begin{enumerate}
    \item \textbf{Phase~1 --- Weekly Scheduling:} Score each candidate job, then assign jobs to (technician, vehicle, day) slots by walking down a priority--sorted list and checking eligibility and capacity rules one by one.
    \item \textbf{Phase~2 --- Daily Stop Ordering:} For each day's assigned job set, arrange stops in a sensible geographic sequence using a nearest--neighbour heuristic.
\end{enumerate}

The engine must:
\begin{itemize}
    \item Serve as many high--value jobs as possible each week, respecting all business rules;
    \item Produce compact, drivable daily routes;
    \item Make every decision explainable---traceable to a named rule and its inputs;
    \item Produce reviewable route/day proposals (not direct calendar writes).
\end{itemize}

% ===================================================================
\section*{Key Objectives (KPIs)}
% ===================================================================
\begin{itemize}
    \item \textbf{Primary goal:} Serve as many high--priority jobs as possible each week (priority = revenue + aging pressure + seasonal fairness).
    \item \textbf{Secondary goals:} Keep daily workloads roughly even across technicians; keep routes geographically compact; maintain area readiness for customer communication.
    \item \textbf{Non--goal for Version~1:} Mathematical optimality. The engine should produce \emph{good, explainable} plans---not provably optimal ones.
\end{itemize}

\section*{System Overview}
% ===================================================================

\begin{itemize}
    \item \textbf{ServiceM8} -- system of record for jobs, technicians, and operational data.
    \item \textbf{Airtable} -- scheduling rules, policy settings, review layers.

The Python engine consumes structured inputs and produces structured outputs; it is independently testable.  Every decision it makes can be traced to a named rule and the data that triggered it.

\section*{Constraints and Parameters}
% ===================================================================

\subsection*{Vehicle Types}
\begin{itemize}
    \item \textbf{V-2} -- required for Radiator Runs (special route type).
    \item \textbf{V-4} -- weekly capacity switch (available / unavailable toggles capacity).
    \item \textbf{V-5, V-6, V-7} -- standard service vehicles.
\end{itemize}

Each vehicle $k$ has a set of capability tags $\mathit{cap}_k$, a daily availability vector, a speed $v_k$ (m/min), a cost rate $c_k$ (\$/m), and a stop capacity $Q_k$.

\subsection*{Technician Parameters}
Each technician $t$ is characterised by:
\begin{itemize}
    \item Skill set $\mathit{skill}_t$.
    \item Weekly availability after exceptions: $\mathcal{D}_t \subseteq \mathcal{D}$.
    \item Home depot / start location $(y_t, x_t)$.
\end{itemize}

\subsection*{Job Parameters}
Each job $j \in \mathcal{J}$ carries:
\begin{itemize}
    \item Address with geocoordinates $(\mathrm{lat}_j, \mathrm{lon}_j)$.
    \item Area label $a_j \in \mathcal{A}$.
    \item Required route type $\rho_j$ (normal, radiator, NH overnight, helper-needed).
    \item Helper flag $h_j \in \{0,1\}$.
    \item Age in days $\alpha_j$ (time since job was created).
    \item Base revenue / value $w_j$.
    \item Service time $\tau_j$ (minutes) -- average on-site duration.
\end{itemize}

\subsection*{Time and Distance}
\begin{itemize}
    \item Cluster radius: $R$ (meters) -- maximum Haversine distance from technician start to any job in the cluster.
    \item Maximum inter-stop distance: $r$ (meters) -- maximum distance between consecutive stops.
    \item Vehicle speed: $v_k$ (m/min) -- depends on vehicle $k \in \mathcal{K}$.
    \item Cost per metre driven: $c_k$ (\$/m) -- depends on vehicle $k \in \mathcal{K}$.
    \item Service time per stop: $\tau_j$ (minutes) -- depends on job $j \in \mathcal{J}$.
    \item Vehicle capacity: $Q_k$ -- maximum number of stops vehicle $k$ can serve per route.
    \item Maximum daily route duration: $T_{\max}$ (minutes).
    \item Maximum stops per route: $P$ (global upper bound; vehicle capacity $Q_k \leq P$).
\end{itemize}

\subsection*{Seasonal Weights}
Scoring weights vary by season:
\[
\text{score}_j \;=\; w_g \cdot \text{geo}_j \;+\; w_f \cdot \text{fair}_j \;+\; w_a \cdot \text{age}_j \;+\; w_r \cdot \text{readiness}_j
\]
where $(w_g, w_f, w_a, w_r)$ shift between a geography--dominant profile (Mar--Sep) and a fairness--dominant profile (Oct--Feb).

\subsection*{Weekly Exceptions}
Before route construction the engine applies weekly exceptions:
\begin{itemize}
    \item Technician unavailability days.
    \item Vehicle unavailability.
    \item Pre-scheduled long jobs and boiler installs that consume partial capacity.
\end{itemize}
These reduce effective weekly capacity before the scheduling layer begins.

\section*{Planning Pipeline}
\label{sec:pipeline}
% ===================================================================

The weekly planning pipeline runs the following stages in order.
Every stage is a deterministic function of its inputs; there is no randomness and no external solver.

\begin{enumerate}
    \item \textbf{Load \& Validate} -- Import candidate pool, technician roster, vehicle roster, exceptions.  Run input-validation checks (Section~\ref{sec:validation}).
    \item \textbf{Exclude} -- Remove jobs already scheduled, cancelled, urgent/manual, or in-workflow.  Emit an exclusion reason code for each removed job.
    \item \textbf{Apply Weekly Exceptions} -- Subtract PTO, maintenance, and pre-booked capacity.  Compute the effective availability sets $\mathcal{D}_t$, $\mathcal{D}_k$.
    \item \textbf{Reserve Special Routes} -- Place Radiator Runs (require V-2), NH Overnight Runs, and helper-needed routes \emph{before} normal filling.
    \item \textbf{Phase~1: Score \& Schedule (Section~\ref{sec:scheduling})} -- Score every remaining job; walk the sorted list and assign each job to the best eligible (technician, vehicle, day) slot, checking rules at every step.
    \item \textbf{Phase~2: Order Stops (Section~\ref{sec:routing})} -- For each day's assigned job set, arrange stops in geographic order using a nearest-neighbour heuristic.  Verify time and distance feasibility; drop jobs that violate constraints.
    \item \textbf{Select Standby Jobs} -- Choose plausible backup jobs per route.
    \item \textbf{Compute Area Readiness} -- Derive area communication layer (\texttt{Good} / \texttt{Moderate} / \texttt{Lean}).
    \item \textbf{Render Outputs} -- Flags, summaries, exclusion report, reason codes.
\end{enumerate}

% ===================================================================
\section*{Phase~1: Rules--Based Weekly Scheduling}
\label{sec:scheduling}
% ===================================================================

The scheduling phase decides \emph{which} jobs to serve and \emph{who does them, with which vehicle, on which day}.
It uses a deterministic score-and-assign loop: no solver, no randomness, fully explainable.

\subsection*{Step 1 --- Job Scoring}

Every candidate job $j \in \mathcal{J}$ receives a composite priority score:
\[
\text{score}_j \;=\; w_g \cdot \text{geo}_j \;+\; w_f \cdot \text{fair}_j \;+\; w_a \cdot \text{age}_j \;+\; w_r \cdot \text{readiness}_j
\]
where:
\begin{itemize}
    \item $\text{geo}_j$: geographic convenience (inverse of average distance from available technicians).
    \item $\text{fair}_j$: fairness component (how under-served the job's area has been recently).
    \item $\text{age}_j$: aging pressure (days since the job was created, $\alpha_j$).
    \item $\text{readiness}_j$: area-readiness bonus (higher if the job's area is currently rated \texttt{Lean}).
\end{itemize}

The four weights $(w_g, w_f, w_a, w_r)$ are set in the config file with two seasonal profiles: geography-dominant (Mar--Sep) and fairness-dominant (Oct--Feb).
The engine selects the active profile based on the planning-week date.

Jobs are sorted by $\text{score}_j$ in descending order.  Ties are broken by $\alpha_j$ (oldest first), then by $w_j$ (highest revenue first).

\subsection*{Step 2 --- Eligibility Check}

For each candidate assignment of job $j$ to technician $t$, vehicle $k$, day $d$, the engine evaluates a named eligibility checklist.  The assignment is \textbf{eligible} only if \emph{every} rule passes:

\medskip
{\small
\begin{longtable}{lp{9.5cm}}
\toprule
\textbf{Rule name} & \textbf{Condition} \\
\midrule
\texttt{TECH\_AVAILABLE}     & Technician $t$ is available on day $d$ after exceptions: $d \in \mathcal{D}_t$. \\
\texttt{VEH\_AVAILABLE}      & Vehicle $k$ is available on day $d$ after exceptions: $d \in \mathcal{D}_k$. \\
\texttt{SKILL\_MATCH}        & Technician skills cover job requirements: $\mathit{skill}_t \supseteq \text{req}(j)$. \\
\texttt{VEH\_CAPABILITY}     & Vehicle capabilities cover route type: $\mathit{cap}_k \supseteq \rho_j$. \\
\texttt{WITHIN\_RADIUS}      & Job is within cluster radius of technician depot: $\text{haversine}(y_t, x_t, \mathrm{lat}_j, \mathrm{lon}_j) \leq R$. \\
\texttt{NOT\_PROHIBITED}     & The (technician, job) pair is not on the prohibited-pairing list. \\
\texttt{CAPACITY\_OK}        & Adding this job would not exceed vehicle stop capacity $Q_k$ for the day. \\
\texttt{TIME\_OK}            & Adding this job's service time $\tau_j$ would not push the day's total past $T_{\max}$ (using a conservative buffer for travel). \\
\texttt{ONE\_VEH\_PER\_TECH} & Technician $t$ has not already been assigned a different vehicle on day $d$. \\
\texttt{ONE\_TECH\_PER\_VEH} & Vehicle $k$ has not already been assigned to a different technician on day $d$. \\
\bottomrule
\end{longtable}
}

If every rule passes, the assignment is feasible.  If any rule fails, the engine records which rule failed and moves to the next candidate slot.

\subsection*{Step 3 --- Slot Selection (Best-Fit Assignment)}

For each job (in score order), the engine considers all eligible (technician, vehicle, day) slots and picks the \textbf{best fit} using a simple preference ranking:

\begin{enumerate}
    \item \textbf{Closest depot.} Among eligible slots, prefer the technician whose home depot is nearest to the job.
    \item \textbf{Lightest day.} Among tied technicians, prefer the day with the least service time already assigned (this naturally balances workloads without a mathematical penalty term).
    \item \textbf{Cheapest vehicle.} Among tied days, prefer the vehicle with the lower cost rate $c_k$.
\end{enumerate}

If no eligible slot exists, the job is \textbf{excluded} with the reason code from the first failing rule (e.g.\ \texttt{SKILL\_MISMATCH}, \texttt{CLUSTER\_RADIUS}).

\subsection*{Step 4 --- Helper Assignment (Two-Man Jobs)}

For every assigned job $j \in \mathcal{J}^H$ (helper-needed), the engine runs a secondary assignment pass to find a helper technician for the same day $d$:

\begin{enumerate}
    \item List all technicians $t' \neq t_{\text{primary}}$ where:
    \begin{itemize}
        \item $d \in \mathcal{D}_{t'}$ (available on the day).
        \item $\mathit{skill}_{t'} \supseteq \text{req}(j)$ (skills match).
        \item $\text{haversine}(y_{t'}, x_{t'}, \mathrm{lat}_j, \mathrm{lon}_j) \leq R$ (within reach).
        \item Adding $\tau_j$ to $t'$'s day does not exceed $T_{\max}$.
    \end{itemize}
    \item Pick the helper with the lightest existing load on day $d$.
    \item If no helper is available, un-assign the job and mark it as \texttt{HELPER\_UNAVAIL}.
\end{enumerate}

\subsection*{Step 5 --- Workload Review}

After all jobs are assigned, the engine checks workload balance:
\begin{itemize}
    \item Compute each technician's total weekly service time.
    \item If any technician exceeds 90\,\% of their available hours, emit a \texttt{TECH\_OVERLOAD} flag.
    \item If any vehicle is at 100\,\% capacity on 4+ days, emit a \texttt{VEH\_BOTTLENECK} flag.
\end{itemize}

No jobs are moved or swapped at this stage---the flags exist so the owner can decide whether the imbalance is acceptable.

\subsection*{Algorithm Summary}

\begin{algorithm}[H]
\caption{Rules--Based Weekly Scheduling (Phase~1)}
\begin{algorithmic}[1]
\Require Candidate jobs $\mathcal{J}$, technicians $\mathcal{T}$, vehicles $\mathcal{K}$, days $\mathcal{D}$, config
\Ensure Assignments list, exclusion list, review flags
\State Compute $\text{score}_j$ for every $j \in \mathcal{J}$
\State Sort $\mathcal{J}$ by score descending (ties: age, then revenue)
\For{each job $j$ in sorted order}
    \State $\text{best\_slot} \gets \textsc{None}$
    \For{each $(t, k, d)$ in eligible slots for $j$}
        \State Check all 10 eligibility rules
        \If{all rules pass \textbf{and} slot is better than $\text{best\_slot}$}
            \State $\text{best\_slot} \gets (t, k, d)$
        \EndIf
    \EndFor
    \If{$\text{best\_slot}$ found}
        \State Assign $j \to \text{best\_slot}$; update capacity trackers
        \If{$j \in \mathcal{J}^H$}
            \State Run helper-assignment pass (Step~4)
        \EndIf
    \Else
        \State Record exclusion: $(j, \text{first failing rule})$
    \EndIf
\EndFor
\State Run workload-review checks; emit flags
\end{algorithmic}
\end{algorithm}

\paragraph{Why this approach.}
\begin{itemize}
    \item \textbf{Explainable:} every assignment can be traced to ``job $j$ scored $X$; slot $(t,k,d)$ was chosen because it was the closest eligible depot on the lightest day.''
    \item \textbf{Deterministic:} same inputs always produce the same plan.
    \item \textbf{Adjustable:} the scoring weights and the slot-preference ranking are config parameters, not hard-coded logic.
    \item \textbf{No solver dependency:} pure Python, no Pyomo, no CBC, no external libraries beyond standard data-processing.
    \item \textbf{Upgrade path:} if the greedy approach leaves too much value on the table, Phase~1 can be replaced with an optimisation model (MILP or constraint programming) in Version~2 without changing the rest of the pipeline.
\end{itemize}

% ===================================================================
\section*{Phase~2: Stop Ordering (Nearest--Neighbour Routing)}
\label{sec:routing}
% ===================================================================

After Phase~1 assigns jobs to (technician, vehicle, day) bundles, Phase~2 arranges the stops within each day into a drivable geographic sequence.
Version~1 uses a simple, explainable heuristic---no VRP solver.

\subsection*{Step 1 --- Build the Day's Stop List}

For each (technician $t$, vehicle $k$, day $d$) bundle, collect the assigned jobs $\mathcal{J}_{t,k,d}$.
The depot (technician start location) is node~0.

\subsection*{Step 2 --- Nearest--Neighbour Sequencing}

\begin{algorithm}[H]
\caption{Nearest--Neighbour Stop Ordering}
\begin{algorithmic}[1]
\Require Depot $(y_t, x_t)$, assigned jobs $\mathcal{J}_{t,k,d}$, distance function $d(\cdot,\cdot)$
\Ensure Ordered route $\Omega$
\State $\text{current} \gets \text{depot}$
\State $\text{remaining} \gets \mathcal{J}_{t,k,d}$
\State $\Omega \gets [\text{depot}]$
\While{remaining $\neq \emptyset$}
    \State $j^* \gets \arg\min_{j \in \text{remaining}} d(\text{current}, j)$
    \State Append $j^*$ to $\Omega$; remove $j^*$ from remaining
    \State $\text{current} \gets j^*$
\EndWhile
\State Append depot to $\Omega$ \Comment{return to start}
\end{algorithmic}
\end{algorithm}

Distances are computed via Haversine (straight-line), which is sufficient for sequencing.
Road-network distances can be substituted without changing the algorithm.

\subsection*{Step 3 --- Feasibility Verification}

After sequencing, the engine walks the route and checks:

\medskip
{\small
\begin{longtable}{lp{9cm}}
\toprule
\textbf{Check} & \textbf{Action if violated} \\
\midrule
Total travel + service time $\leq T_{\max}$ & Drop the last-added job from the route; re-check.  Dropped job gets reason code \texttt{ROUTE\_DROP}. \\
Number of stops $\leq Q_k$ & Should not happen (enforced in Phase~1), but drop excess if it does. \\
Inter-stop distance $\leq r$ & Flag the arc as \texttt{CROSS\_AREA}; do not drop (guidance, not hard wall). \\
Every stop within radius $R$ of depot & Should not happen (eligibility enforced), but flag \texttt{GEOCODE\_OOB} if violated. \\
\bottomrule
\end{longtable}
}

If any job is dropped, the engine re-runs the nearest-neighbour on the reduced set so the sequence stays sensible.

\subsection*{Why nearest-neighbour (not a solver)}
\begin{itemize}
    \item \textbf{Transparent:} the route order is ``go to the closest un-visited stop.''  No parameter tuning, no black-box behaviour.
    \item \textbf{Fast:} $O(n^2)$ for $n$ stops.  For typical NES clusters ($n \leq 14$), this is instantaneous.
    \item \textbf{Good enough:} nearest-neighbour typically delivers routes within 20--25\,\% of the optimal tour length.  For Version~1 the priority is correctness and reviewability, not distance optimality.
    \item \textbf{Upgrade path:} if route quality matters (fuel costs, tight schedules), a 2-opt improvement pass or a proper TSP/VRP solver can be added in Version~2 as a drop-in replacement.
\end{itemize}

% ===================================================================
\section*{Hard Constraints Summary}
% ===================================================================

The following rules are enforced throughout the pipeline.  None may be relaxed; every violation either blocks the assignment or triggers a job exclusion with a reason code.

\begin{itemize}
    \item Job must be in the valid weekly candidate pool (\texttt{status = candidate}).
    \item Technician must be available on the planned day (after weekly exceptions): \texttt{TECH\_AVAILABLE}.
    \item Vehicle must be available and capability-compatible with the route type: \texttt{VEH\_AVAILABLE}, \texttt{VEH\_CAPABILITY}.
    \item Technician skills must cover job requirements: \texttt{SKILL\_MATCH}.
    \item Job must be within cluster radius $R$ of the technician's depot: \texttt{WITHIN\_RADIUS}.
    \item Prohibited technician--job pairings are excluded: \texttt{NOT\_PROHIBITED}.
    \item Vehicle stop capacity $Q_k$ must not be exceeded: \texttt{CAPACITY\_OK}.
    \item Daily service-time budget $T_{\max}$ must not be exceeded: \texttt{TIME\_OK}.
    \item Each technician uses at most one vehicle per day: \texttt{ONE\_VEH\_PER\_TECH}.
    \item Each vehicle is used by at most one technician per day: \texttt{ONE\_TECH\_PER\_VEH}.
    \item Helper-needed jobs require exactly one helper technician, different from the primary.
    \item No job is assigned more than once.
\end{itemize}

Each constraint maps directly to a named rule in the eligibility-check table (Phase~1, Step~2) or the feasibility-verification table (Phase~2, Step~3).

\section*{Special Route Types}
% ===================================================================

Special routes are planned \emph{before} normal route filling to reserve capacity:

\begin{enumerate}
    \item \textbf{Radiator Runs} -- Require vehicle V-2. Scheduled first.
    \item \textbf{NH Overnight Runs} -- Long-distance routes requiring overnight stay.
    \item \textbf{Two-Man / Helper Routes} -- Jobs flagged with $h_j = 1$; need two technicians or a helper.
\end{enumerate}

After special routes consume capacity, the remaining capacity feeds normal scheduling.

% ===================================================================
\section*{Solution Approach}
% ===================================================================

Version~1 uses a single, deterministic algorithm for each phase:

\begin{itemize}
    \item \textbf{Phase~1 (Scheduling):} Score--sort--assign greedy loop with 10 named eligibility rules (Section~\ref{sec:scheduling}).
    \item \textbf{Phase~2 (Routing):} Nearest--neighbour stop ordering with post-hoc feasibility checks (Section~\ref{sec:routing}).
\end{itemize}

No external solver, optimisation library, or metaheuristic is required.
The engine completes a full weekly plan for the current fleet size (30 jobs, 4 technicians, 5 vehicles, 5 days) in under one second on commodity hardware.

If route quality improvements are needed in a future version, a 2-opt improvement pass or a TSP/VRP solver can be introduced as a drop-in replacement for the nearest-neighbour step without changing the rest of the pipeline.

% ===================================================================
\section*{Standby Job Selection}
% ===================================================================

After route construction, unassigned but feasible jobs are ranked as standby replacements for each route. Standby jobs are output separately from assigned jobs so they can be deployed if a scheduled job drops out.

% ===================================================================
\section*{Area Readiness}
% ===================================================================

The engine computes an area--readiness indicator for each geographic area $a \in \mathcal{A}$:

\[
\text{Readiness}(a) \in \{\texttt{Good},\;\texttt{Moderate},\;\texttt{Lean}\}
\]

based on the density and quality of planned activity in that area. This is a communication layer for customer--facing scheduling messages and does not drive route construction.

% ===================================================================
\section*{Post-Route Update Rule}
% ===================================================================

After a weekly plan is finalised and approved, assigned jobs are marked as \texttt{scheduled} and removed from the candidate pool for subsequent planning iterations.

% ===================================================================
\section*{Output Artefacts}
% ===================================================================

Each weekly run produces:
\begin{itemize}
    \item Proposed route/day assignments per technician.
    \item Standby job list per route.
    \item Plotted map for each route/day.
    \item Area readiness values and plain-text lookup block.
    \item Review flags: duplicate-address groups, vehicle bottlenecks, cross-area blends, weak standby pools.
    \item Reason codes traceable to the rules that produced each decision.
\end{itemize}

Outputs are designed for human review by a non-technical owner; the engine does \emph{not} auto-book jobs.

% ===================================================================
\section*{Owner--Controlled Policy Inputs}
% ===================================================================

The engine exposes a compact set of tunable parameters that a non-technical business owner can adjust---via Airtable fields or a configuration file---to steer planning behaviour without modifying code.

\paragraph{Maximum daily route duration ($T_{\max}$).}
Caps the total travel-plus-service time for any single route.
Lowering $T_{\max}$ produces shorter days and leaves more buffer for delays; raising it allows more jobs per day but increases overtime risk.

\paragraph{Maximum stops per route ($P$) and vehicle capacities ($Q_k$).}
$P$ is a global upper bound; each vehicle's capacity $Q_k \leq P$ can be set independently.
Reducing $Q_k$ for a specific vehicle reserves slack for ad-hoc stops or equipment space.

\paragraph{Cluster radius ($R$) and inter-stop limit ($r$).}
$R$ defines how far from a technician's home depot jobs may be included; $r$ limits the maximum gap between consecutive stops.
Tightening these values produces geographically compact routes with shorter drive legs; relaxing them allows the engine to reach more jobs at the expense of longer routes.

\paragraph{Seasonal weight profile $(w_g, w_f, w_a, w_r)$.}
Four weights that shift the job-scoring formula between a geography-dominant mode (Mar--Sep) and a fairness-dominant mode (Oct--Feb).
The owner sets two seasonal profiles; the engine selects the active profile based on the planning-week date.

\paragraph{Weekly exceptions.}
Technician and vehicle unavailability entries (PTO, maintenance, pre-booked installs) entered each week.
These directly shrink the effective capacity seen by Phase~1 and must be finalised before the planning run.

\paragraph{Special-route reservations.}
The owner marks which jobs require Radiator Run (V-2), NH Overnight, or helper-needed handling.
Reserved routes are locked before normal filling, so the owner controls how much capacity is set aside for these categories.

\paragraph{Prohibited pairings and area definitions.}
Technician--job exclusions (e.g.\ customer-requested) and the geographic area map ($\mathcal{A}$) are maintained in Airtable.
Changes take effect on the next planning run without engine modification.

\section*{Major Risks and Ambiguities}
% ===================================================================

\paragraph{Phase-1 / Phase-2 feasibility gap.}
Phase~1 budgets service time only; it does not account for travel time between stops.
Consequently, some bundles passed to Phase~2 may exceed $T_{\max}$ once driving distances are added, forcing jobs to be dropped at the routing stage.
A conservative $T_{\max}$ buffer (e.g.\ 80\,\% of the true day length) should be used in Phase~1 scoring until an iterative feedback loop is added.

\paragraph{Distance-matrix quality.}
Road-network distances are computed via OSMnx/NetworkX, with a Haversine fallback when no path exists.
If the underlying OpenStreetMap data is incomplete or stale for rural New England roads, route durations may be under- or over-estimated.
Periodic comparison with Google Maps drive-time queries is recommended.

\paragraph{Greedy scheduling quality.}
The score--sort--assign loop is deterministic but greedy: it processes jobs in priority order and cannot backtrack.
A high-priority job may consume a slot that would have been better used by two medium-priority jobs.
Monitoring the fraction of unassigned jobs and the fill-rate variance across technicians will reveal whether the greedy strategy needs iterative improvement (e.g.\ swap passes) in a future version.

\paragraph{Geocoding accuracy.}
Job coordinates originate from ServiceM8 address geocoding.
Incorrect or ambiguous addresses produce wrong cluster assignments and distance estimates.
A pre-run validation step that flags coordinates outside known service territory is advisable.

\paragraph{Exception staleness.}
Weekly exceptions must be entered before the planning run.
Late-arriving changes (e.g.\ same-day sick leave) are not captured and require manual route adjustment or a re-run, which the current pipeline does not automate.

\paragraph{Seasonal-weight calibration.}
The scoring weights $(w_g, w_f, w_a, w_r)$ are currently set by judgement rather than historical data.
Until job-outcome data is available for back-testing, the weights should be reviewed periodically and adjusted if dispatchers consistently override engine recommendations.

\paragraph{Helper-job coordination.}
Two-man jobs ($h_j = 1$) are assigned a helper in Phase~1, Step~4.
However, the model assumes the helper travels with the primary technician's vehicle and does not account for the helper's own travel time to the job site.
If helpers must drive separately, an additional travel-feasibility check is needed.

\paragraph{Single-week planning horizon.}
Each run plans one week in isolation.
Jobs deferred this week re-enter the pool next week with a higher age score, but there is no explicit multi-week look-ahead to prevent systematic starvation of low-value areas.

\section*{Recommended Engine Structure}
\label{sec:engine_structure}
% ===================================================================

The Python engine is organised as a pipeline of self-contained stages, each with explicit inputs, outputs, and a single responsibility.
No external solver or optimisation library is required.
The recommended package layout is:

\begin{verbatim}
nes_dispatch/
    __init__.py
    data/
        loaders.py          # CSV / Airtable readers -> dataclasses
        models.py           # Job, Technician, Vehicle, WeeklyException, WeeklyData
        validators.py       # Pre-run sanity checks (coordinates, missing fields)
    rules/
        scoring.py          # Composite priority-score formula
        eligibility.py      # 10 named eligibility rules -> pass/fail per slot
        slot_selection.py   # Best-fit ranking (closest depot, lightest day, cheapest vehicle)
        helpers.py          # Helper-assignment secondary pass
        workload.py         # Workload-review flags (no job movement)
    routing/
        distance.py         # OSMnx / Haversine distance computation
        nearest_neighbour.py# NN stop ordering + feasibility verification
    postprocess/
        standby.py          # Standby job ranking
        readiness.py        # Area readiness computation
        output.py           # Route maps, text summaries, reason codes
    config.py               # Loads JSON/YAML config; merges Airtable overrides
    pipeline.py             # Top-level orchestrator (the 9-step pipeline)
\end{verbatim}

\paragraph{Design principles.}
\begin{enumerate}
    \item \textbf{Stateless stages.} Each stage is a pure function: it receives typed dataclass inputs and returns typed outputs. No global mutable state.
    \item \textbf{Fail-fast validation.} \texttt{validators.py} runs before the pipeline and rejects runs with missing columns, out-of-bounds coordinates, or conflicting exceptions. Errors are collected, not thrown one at a time.
    \item \textbf{Named rules.} Every scheduling decision is traceable to a named rule (e.g.\ \texttt{SKILL\_MATCH}, \texttt{CAPACITY\_OK}) and the inputs that triggered it. No opaque scoring or black-box behaviour.
    \item \textbf{Idempotent runs.} Given the same inputs and config, the engine produces identical outputs. The algorithm is fully deterministic---no random seeds, no stochastic components.
    \item \textbf{Separation of I/O.} File reads, API calls, and map rendering live in \texttt{data/loaders.py} and \texttt{postprocess/output.py}. Core logic never touches the filesystem directly.
\end{enumerate}

\paragraph{Pipeline orchestration (\texttt{pipeline.py}).}
The top-level function \texttt{run\_weekly\_plan(config\_path, data\_dir)} executes the nine stages listed in Section~\ref{sec:pipeline} in sequence.
Each stage logs its runtime and emits a structured summary (JSON) so that failures can be diagnosed without re-running the full pipeline.

\paragraph{Testing strategy.}
Unit tests cover individual stages with small synthetic fixtures.
An integration test runs the full pipeline on the example CSVs and asserts that:
\begin{itemize}
    \item every assigned job satisfies all 10 eligibility rules;
    \item no technician exceeds $T_{\max}$;
    \item helper jobs have exactly one primary and one distinct helper;
    \item output files are generated without errors.
\end{itemize}

\section*{Code vs.\ Configuration vs.\ Airtable Split}
\label{sec:split}
% ===================================================================

Not all system behaviour belongs in code.
The following table defines which concerns live in Python source, which in a version-controlled config file (JSON/YAML), and which in Airtable (editable by the owner at any time).

\medskip
{\small
\begin{longtable}{p{5.2cm}ccc}
\toprule
\textbf{Concern} & \textbf{Code} & \textbf{Config} & \textbf{Airtable} \\
\midrule
Rule-based scheduling logic                   & $\bullet$ &           &           \\
Nearest-neighbour routing                     & $\bullet$ &           &           \\
Eligibility-check rules                       & $\bullet$ &           &           \\
Distance computation                          & $\bullet$ &           &           \\
Output rendering (maps, text)                 & $\bullet$ &           &           \\
\midrule
$T_{\max}$ (daily time budget)                &           & $\bullet$ &           \\
$P$ (global max stops)                        &           & $\bullet$ &           \\
$R$ (cluster radius)                          &           & $\bullet$ &           \\
$r$ (inter-stop radius)                       &           & $\bullet$ &           \\
Seasonal weight profiles $(w_g,w_f,w_a,w_r)$  &           & $\bullet$ &           \\
Workload-review thresholds                    &           & $\bullet$ &           \\
\midrule
Vehicle roster \& capability tags             &           &           & $\bullet$ \\
Vehicle availability days                     &           &           & $\bullet$ \\

Special-route flags (radiator, NH, helper)    &           &           & $\bullet$ \\
Job candidate pool (from ServiceM8 sync)      &           &           & $\bullet$ \\
\bottomrule
\end{longtable}
}

\paragraph{Rationale.}
\begin{itemize}
    \item \textbf{Code} holds logic that requires developer review to change--model structure, algorithm implementations, output formats. Changes go through version control and testing.
    \item \textbf{Config (JSON/YAML)} holds numerical parameters that a developer or technically confident owner adjusts between runs. The config file is version-controlled alongside the code so that every run is reproducible. A schema validator rejects unknown keys or out-of-range values before the pipeline starts.
    \item \textbf{Airtable} holds operational data that changes weekly or on demand--rosters, exceptions, pairings, area maps. The owner edits these freely; the engine pulls the latest state at the start of each run via the Airtable API. No code deployment is needed.
\end{itemize}

\paragraph{Override precedence.}
When the same parameter appears in both the config file and an Airtable field (e.g.\ a per-vehicle capacity override), the Airtable value takes precedence.
This lets the owner make one-off adjustments without touching the config file, while the config file provides sensible defaults.

\paragraph{Versioning and audit trail.}
Every planning run records:
\begin{enumerate}
    \item The config-file hash (git commit or content hash).
    \item A snapshot of the Airtable tables consumed (written to a timestamped JSON file).
    \item The engine version (Python package version or git SHA).
\end{enumerate}
This triple ensures that any past run can be fully reproduced or audited.

\section*{Validation and Testing Approach}
\label{sec:validation}
% ===================================================================

Correctness is enforced at three levels: input validation (before the pipeline runs), constraint verification (after each layer), and regression testing (during development).

\subsection*{Input Validation (Pre-run)}

\texttt{validators.py} performs the following checks when data is loaded. If any check fails, the run is aborted with a consolidated error report--not a single exception.

\begin{enumerate}
    \item \textbf{Schema completeness.} Every required CSV column is present and non-null for every row.
    \item \textbf{Coordinate bounds.} All latitudes fall within $[41, 48]$ and longitudes within $[-74, -67]$ (approximate New England bounding box). Out-of-range values indicate geocoding errors.
    \item \textbf{Referential integrity.} Every \texttt{scope\_id} in the exceptions file matches an existing technician or vehicle ID. Every \texttt{area\_id} on a job exists in the area definitions.
    \item \textbf{Availability consistency.} After applying exceptions, at least one (technician, vehicle, day) triple must remain feasible; otherwise the run would produce an empty schedule.
    \item \textbf{Duplicate detection.} Duplicate \texttt{job\_id}, \texttt{tech\_id}, or \texttt{vehicle\_id} values are flagged.
    \item \textbf{Config schema.} The JSON/YAML config is validated against a schema: all numerical parameters are present, positive, and within documented bounds (e.g.\ $T_{\max} > 0$, $R > 0$, $P \geq 1$).
\end{enumerate}

\subsection*{Post-Phase Constraint Verification}

After Phase~1 and Phase~2 each complete, the engine runs an assertion suite on the solution before proceeding.

\paragraph{Phase~1 assertions.}
\begin{itemize}
    \item Every assigned job appears at most once across all (technician, vehicle, day) triples.
    \item All 10 eligibility rules pass for every assignment.
    \item Vehicle capacity is satisfied per slot: assigned jobs $\leq Q_k$.
    \item Service-time budget holds: total service time $\leq T_{\max}$ per slot.
    \item Each technician uses at most one vehicle per day.
    \item Each vehicle is used by at most one technician per day.
    \item Every helper job that is assigned has exactly one helper technician, different from the primary.
\end{itemize}

\paragraph{Phase~2 assertions (per route).}
\begin{itemize}
    \item Route departs from and returns to the depot.
    \item No stop is visited more than once.
    \item Total travel-plus-service time $\leq T_{\max}$.
    \item Number of visited stops $\leq Q_k$.
    \item All stops lie within radius $R$ of the depot.
\end{itemize}

If any assertion fails, the pipeline logs the violation, marks the affected route as \texttt{INFEASIBLE}, and continues with the remaining bundles rather than aborting the entire run.

\subsection*{Regression and Unit Testing}

\paragraph{Unit tests.}
Each module exposes pure functions that can be tested in isolation:
\begin{itemize}
    \item \texttt{test\_scoring}: synthetic 5-job scenario; verify composite scores match hand-computed values.
    \item \texttt{test\_eligibility}: 3-job, 2-tech, 2-vehicle scenario; verify each of the 10 named rules returns the expected pass/fail.
    \item \texttt{test\_slot\_selection}: given pre-computed eligibility, verify best-fit ranking (closest depot $\to$ lightest day $\to$ cheapest vehicle).
    \item \texttt{test\_nearest\_neighbour}: 5-node instance with known optimal NN tour; compare output sequence.
    \item \texttt{test\_feasibility}: route deliberately exceeding $T_{\max}$; verify job-drop and re-sequence logic.
    \item \texttt{test\_helper}: 2-tech, 1 helper job; confirm that exactly one helper is assigned and differs from the primary.
\end{itemize}

\paragraph{Integration test.}
A single end-to-end test runs the full pipeline on the example CSVs (\texttt{data/example\_*.csv}) and asserts:
\begin{itemize}
    \item no validation errors;
    \item at least one route is generated;
    \item all post-phase assertions pass;
    \item output files (maps, summaries, reason codes) are created and non-empty.
\end{itemize}

\paragraph{Golden-file comparison.}
Because the engine is fully deterministic (no random components), every run on the same inputs produces identical outputs.
The integration test compares output against a checked-in golden file; a mismatch triggers a diff report so that unintended behavioural changes are caught immediately.

\section*{Review Flags and Exception Outputs}
\label{sec:flags}
% ===================================================================

The engine produces a structured set of review flags alongside every weekly plan.
These flags are designed for the non-technical owner to scan quickly and decide whether to approve, adjust, or re-run.

\subsection*{Flag Categories}

Each flag has a severity (\texttt{INFO}, \texttt{WARN}, \texttt{CRITICAL}), a short code, and a human-readable message.

\medskip
{\small
\begin{longtable}{llp{7.5cm}}
\toprule
\textbf{Code} & \textbf{Severity} & \textbf{Meaning} \\
\midrule
\texttt{DUP\_ADDR}    & INFO     & Two or more jobs share the same address. Grouped for route evaluation; review for possible duplicates. \\
\texttt{CROSS\_AREA}  & INFO     & A route mixes jobs from two or more areas. May be optimal geographically but warrants a glance. \\
\texttt{WEAK\_STANDBY} & WARN    & Fewer than 2 feasible standby jobs exist for a route. If a scheduled job drops, coverage is thin. \\
\texttt{VEH\_BOTTLENECK} & WARN  & A vehicle is assigned at 100\,\% capacity on 4+ days. Little room for ad-hoc or emergency jobs. \\
\texttt{TECH\_OVERLOAD} & WARN   & A technician's total weekly service time exceeds 90\,\% of their available hours. \\
\texttt{HELPER\_TRAVEL} & WARN   & The helper technician's home depot is more than $R$ metres from the primary technician's depot. Travel time may be underestimated. \\
\texttt{ROUTE\_DROP}  & WARN     & One or more Phase-1-scheduled jobs were dropped during Phase~2 routing because travel time pushed the route past $T_{\max}$. \\
\texttt{GEOCODE\_OOB} & CRITICAL & A job's coordinates fall outside the New England bounding box. Likely a geocoding error; the job is excluded and must be corrected. \\
\texttt{NO\_FEASIBLE} & CRITICAL & A (technician, vehicle, day) bundle produced by Phase~1 has no feasible route at all in Phase~2. The entire bundle is unserved. \\
\texttt{SOLVER\_TIMEOUT} & \multicolumn{2}{p{8.5cm}}{\emph{Removed in Version~1---no solver is used.  Reserved for future Version~2 if a TSP/VRP solver is introduced.}} \\
\bottomrule
\end{longtable}
}

\subsection*{Exception / Exclusion Report}

For every job in the weekly candidate pool that is \emph{not} assigned to a route, the engine emits a reason code explaining why.
This report is the primary traceability artefact.

\medskip
{\small
\begin{longtable}{lp{9cm}}
\toprule
\textbf{Reason Code} & \textbf{Explanation} \\
\midrule
\texttt{ALREADY\_ASSIGNED}   & Job was assigned in a prior run and has not been reset. \\
\texttt{STATUS\_EXCLUDED}    & Job status is \texttt{excluded} or \texttt{cancelled} in ServiceM8. \\
\texttt{NO\_ELIGIBLE\_TRIPLE}& No (technician, vehicle, day) triple satisfies all 10 eligibility rules. \\
\texttt{CAPACITY\_FULL}      & All eligible slots were already at vehicle capacity $Q_k$. \\
\texttt{TIME\_BUDGET}        & Including this job would exceed the service-time budget $T_{\max}$ in every eligible slot. \\
\texttt{CLUSTER\_RADIUS}     & Job is outside the cluster radius $R$ from every available technician's depot. \\
\texttt{SKILL\_MISMATCH}     & No available technician has the required skills. \\
\texttt{VEHICLE\_MISMATCH}   & No available vehicle has the required capability tags (e.g.\ radiator job but V-2 unavailable). \\
\texttt{HELPER\_UNAVAIL}     & Job requires a helper ($h_j = 1$) but no second technician is eligible on any feasible day. \\
\texttt{PROHIBITED\_PAIR}    & All eligible technicians are on the prohibited-pairing list for this job. \\
\texttt{ROUTE\_DROP}         & Job was scheduled by Phase~1 but dropped during Phase~2 routing due to travel-time infeasibility. \\
\bottomrule
\end{longtable}
}

\subsection*{Output Format}

Flags and exclusion reasons are written to two files per run:

\begin{itemize}
    \item \texttt{flags\_\{date\}.json} -- array of flag objects, each with \texttt{code}, \texttt{severity}, \texttt{message}, and references to the affected route, technician, or job.
    \item \texttt{exclusions\_\{date\}.json} -- array of exclusion objects, each with \texttt{job\_id}, \texttt{reason\_code}, and a human-readable \texttt{detail} string.
\end{itemize}

Both files are included in the review packet sent to the owner alongside route maps and summaries.
\texttt{CRITICAL} flags are additionally surfaced as bold-red banners at the top of the summary output, ensuring they are not overlooked.

\section*{Future Considerations (Version~2 and Beyond)}
% ===================================================================

\begin{itemize}
    \item \textbf{Iterative Phase-1/Phase-2 feedback:} re-schedule jobs dropped by Phase~2 into alternative slots.
    \item \textbf{2-opt improvement pass:} after nearest-neighbour sequencing, apply pairwise arc swaps to shorten routes.  Low complexity, high payoff.
    \item \textbf{TSP/VRP solver (optional upgrade):} replace the nearest-neighbour step with a proper solver (e.g.\ OR-Tools, Pyomo/CBC) for provably optimal stop ordering.  The rest of the pipeline remains unchanged.
    \item \textbf{MILP scheduling (optional upgrade):} if the greedy scheduler proves too limited, formulate Phase~1 as a mixed-integer program.  The named rules become solver constraints; the upgrade is additive, not a rewrite.
    \item \textbf{Exact appointment-time optimisation:} assign specific arrival windows per stop once route ordering is reliable.
    \item \textbf{Real-time traffic / drive-time API integration:} replace Haversine or OSMnx distances with live data.
    \item \textbf{Historical job-outcome data:} feed completion rates and no-show rates back into the scoring formula.
    \item \textbf{Multi-week rolling horizon planning:} prevent systematic starvation of low-value areas.
    \item \textbf{ML-assisted demand forecasting:} proactive area readiness based on predicted job volume.
\end{itemize}
